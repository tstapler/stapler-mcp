# Validation Plan: docs-index

**Date**: 2026-07-14

## Happy Path Scenario
Given a running `stapler-mcp` daemon with no sources indexed yet, when the user calls `index_docs` with a seed URL (e.g. `https://tokio.rs/tokio/tutorial`), then `search_docs` against that source returns ranked, relevant chunks, `list_indexed_sources` shows the new source with correct counts, and `remove_indexed_source` deletes it so a subsequent `search_docs` call fails with an unknown-source error — proving the full crawl → chunk → embed → store → rank → enumerate → delete lifecycle works end-to-end without a Node process. *(Anchors Story 6.2.1's `docs_index_round_trip` integration test and requirements.md's Success Criteria.)*

## UX Acceptance Tests
N/A — docs-index has no user-facing GUI surface. It is an MCP tool consumed by an LLM coding agent; tool ergonomics (naming, error messages, response shapes) were validated in Phase 2's `research/ux.md` and encoded directly into plan.md's schema/error-handling tasks (e.g. Story 4.2.1's unknown-source and embedding-model-mismatch error message shapes, Story 5.2.1's tool descriptions) rather than as separate human-verifiable UX acceptance criteria.

## Migration Test
N/A — plan.md's Migration Plan section states this is a "Greenfield feature — no existing `~/.stapler-mcp/docs-index/` data to migrate," so no reversible-migration test applies. The one forward-looking consistency check (a source's stored `embedding_model` no longer matching the daemon's current `EMBEDDING_MODEL_ID`) is already covered above under REQ-5's `should_return_model_mismatch_error_without_scoring_when_stored_embedding_model_differs_from_current` test — this is a data-consistency guard, not a migration path (no automatic re-embedding exists or is planned for v1).

## Test Stack
- **Unit**: `cargo test` + built-in `assert!`/`assert_eq!`/`assert!((a - b).abs() < 1e-6)` for float comparisons, following the existing `#[cfg(test)] mod tests` pattern already used in `crates/core` (e.g. `crates/core/src/tools/webcrawl.rs`) and `crates/native`/`crates/cli` — no external test framework (no `rstest`, no `proptest`) is used anywhere else in this codebase, so none is introduced here.
- **Integration**: two distinct flavors, both already established by this codebase:
  1. **Core-crate-level, in-process integration**: `docs.rs`'s own functions (`index_source`, `search_docs`, `remove_indexed_source`) tested against test-double ports — `FakeHttpClient` (fixed URL→response map, matching the style implied by plan.md's Task 4.1.1c), `InMemoryFileStore` (an in-memory `FileStore` impl), and `FakeEmbedder` (deterministic fixed-dimension vectors, e.g. `vec![0.0f32; 384]` per the trait's own doc-comment example in Story 2.1.1's acceptance criterion). None of these fakes exist yet in the codebase (docs-index is the first feature to need them) — they should be added alongside the first tests that need them in `crates/core/src/tools/docs.rs`'s test module.
  2. **Real daemon-spawning integration**: the established pattern in `crates/cli/tests/daemon_ping.rs`/`crates/cli/tests/webcrawl.rs` — build the real binary, spawn it, drive it over the real Unix-socket protocol via `stapler_mcp_core::client::call`, using a `TestEnv: EnvPort` for an isolated `STAPLER_MCP_HOME` tempdir and a hand-rolled mock HTTP server (`spawn_mock_site`-style) for network calls. `crates/cli/tests/docs_index.rs` (Story 6.2.1) follows this exact pattern, but is `#[ignore]`d by default because `NativeEmbedder` downloads a real ~90MB ONNX model from Hugging Face Hub on first use — too slow/flaky for a default `cargo test` run, matching no existing precedent in this codebase (every other integration test here is fully offline/hermetic), which is why it's the one deliberately-opted-out exception.
- **E2E / UX**: N/A (see above) — but the Phase 6 `#[ignore]`d `docs_index_round_trip` test (Story 6.2.1) is the closest analog to an E2E check: it is the only test in this plan that exercises the real embedding model, the real daemon socket, and all four tools together in one call sequence, matching this project's existing informal-canary approach ("the first real-world `index_docs` call... doubles as a manual smoke test" — Risk Control, Staged rollout) rather than a formal E2E suite.
