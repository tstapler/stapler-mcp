# Pre-mortem: docs-index
**Date**: 2026-07-14

## Failure Modes

| # | Failure | First Symptom | Prevention | Severity |
|---|---------|--------------|------------|----------|
| 1 | `all-MiniLM-L6-v2` is a general-purpose sentence-similarity model trained mostly on natural-language sentence pairs (NLI/paraphrase corpora), not code/API-reference text — it may rank Rust crate docs (dense with identifiers like `tokio::spawn`, generic bounds, type signatures) poorly, so `search_docs` "works" (no crash, passes every unit/integration test) but returns mediocre or oddly-ranked results for exactly the code-heavy content that's Tyler's primary real-world use case, and he quietly stops using it in favor of grep/the old tool. | Query for a specific API (e.g. "how do I spawn a task") returns generically-relevant-but-wrong chunks, or top-k scores cluster tightly (e.g. 0.3–0.5) with no clear separation between relevant and irrelevant chunks — a signal no automated test catches, since `cosine_similarity` is unit-tested with hand-picked orthogonal/near vectors and the one real-model integration test (Story 6.2.1) only asserts "at least 1 result," never relevance quality. | Add a small relevance-eval step to `validation.md` before sign-off: index one real Rust crate doc source, run 4–5 known queries with an expected top-3 chunk in mind, and manually confirm results are actually useful (not just non-empty) — plus record in ux.md/plan an explicit escape hatch (e.g. a lexical/substring fallback mode, or hybrid BM25+embedding scoring) to reach for if quality disappoints in practice, rather than treating "ships and returns results" as the definition of done. | P1 |
| 2 | `search_docs`/`list_indexed_sources` are the same tool names already exposed by the existing, currently-connected `docs-mcp-server` MCP integration (`mcp__docs__search_docs`, `mcp__docs__list_libraries`, etc. — confirmed live in this very session's tool list) — the plan's Migration Plan section only addresses greenfield data migration, not decommissioning or disambiguating from the still-running old tool, so during the (unplanned-for) coexistence period an LLM caller can invoke the wrong same-named tool and get a confusing "no indexed source" error, or Tyler ends up indexing the same doc source twice into two separate systems. | Claude Code calls `search_docs` and gets an unexpected unknown-source error because the source was indexed in the *other* docs tool, or Tyler notices duplicate indexing effort across both systems. | Add an explicit cutover/decommission task to the plan (Risk Control or Migration Plan section): once `docs-index` is verified working, remove/disable the `"docs"` MCP server entry from `~/.claude.json` (per `requirements.md`'s own stated goal of "no separate Node process needed at all") rather than leaving both tools live indefinitely with identical names. | P2 |
| 3 | `chunk_markdown`'s `text-splitter::MarkdownSplitter` splits at a `200..800`-character budget on heading/paragraph boundaries, but Story 3.2.1's acceptance criteria only test prose sections — nothing verifies behavior on fenced code blocks (extremely common in Rust crate docs), so a code example can be split mid-fence, producing a chunk with an unclosed/truncated code block that's both a worse embedding input and, more visibly, broken/truncated code shown directly in `search_docs`'s `text` output. | A `search_docs` result's `text` field contains a code snippet that starts or ends mid-line, or is missing its closing ` ``` ` fence — useless when read or pasted back into a conversation. | Add explicit test cases to Story 3.2.1 / `validation.md` with markdown containing fenced code blocks that straddle the chunk-size boundary, and verify (or fix) that `chunk_markdown` never splits inside a ` ``` ` fence, even if that means exceeding the 800-char budget for that one chunk. | P2 |
| 4 | The entire design (brute-force cosine, one JSONL file per source, no eviction/TTL, no total-size accounting anywhere in the wire schema) is explicitly built for "a handful of doc sources," but nothing prevents that number from growing once the tool is convenient and frictionless — and no tool response (`list_indexed_sources`, `index_docs`) ever reports total on-disk bytes used, so growth is invisible until `~/.stapler-mcp/docs-index/` is unexpectedly large or disk space becomes a problem elsewhere. | `df`/disk-usage alerts fire, or Tyler notices `~/.stapler-mcp` has grown to multiple GB, months after the fact — there's no earlier signal because `list_indexed_sources`'s `chunkCount`/`pageCount` fields don't translate to a human-legible size, and nothing warns as the assumption's scale bound is crossed. | Add a cheap `totalBytesOnDisk` (or per-source `bytesOnDisk`) field to `list_indexed_sources`'s output (sum of `chunks.jsonl`/`meta.json` file sizes — already available via `FileStore`/filesystem metadata, no new port needed) so growth is visible the next time the tool is used, even without a dedicated eviction feature. | P2 |
| 5 | Only the *indexing* path's inline-blocking cost was benchmarked and bounded (`MAX_CHUNKS_PER_SOURCE` via Task 2.1.2c) — the *search* path (`search_docs`) has no equivalent benchmark or bound: every call reads and line-by-line JSON-parses the *entire* `chunks.jsonl` for a source (potentially up to `MAX_CHUNKS_PER_SOURCE`, i.e. several hundred to a couple thousand full 384-float-vector records) and scores every one, synchronously, on every single search — and searches happen far more often and more interactively than the rare, explicit `index_docs` call the 8-second budget was calibrated for. | `search_docs` feels noticeably laggy specifically on the largest/most-indexed sources — ironic because those are exactly the sources Tyler cares most about and uses most, so success at adoption makes the problem worse over time. | Benchmark `search_docs`'s read+parse+score path the same way Task 2.1.2c benchmarked `embed()` throughput, and record an expected latency ceiling at `MAX_CHUNKS_PER_SOURCE` scale in `validation.md`; if it's not comfortably sub-second, consider an in-memory per-source embeddings cache (avoid re-parsing the full `chunks.jsonl` on every search of a "hot" source) as a fast-follow rather than discovering the regression only after it's already annoying in daily use. | P2 |

## P1 Items (address before implementation)
- [x] #1 — **Addressed in plan.md**: added Epic 6.3 / Story 6.3.1 / Task 6.3.1a — a manual
  relevance-quality spot check (real Rust doc source, 3-5 realistic queries, human-judged top-1/2
