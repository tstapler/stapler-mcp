## Blocker Resolution Status
- [x] Blocker 1 (`MAX_CHUNKS_PER_SOURCE`) — RESOLVED — Task 2.1.2c (`plan.md:582-616`) replaces the guess with a concrete, executable benchmark: a `#[test] #[ignore]`d test or scratch binary embeds 100 representative 200–800-char chunks through the real `NativeEmbedder`, times it with `std::time::Instant`, computes `chunks_per_second`, and sets `MAX_CHUNKS_PER_SOURCE = floor(chunks_per_second × 8)`. The 8-second budget is explicitly justified (Pattern Decisions row, `plan.md:107`, and inline in Task 2.1.2c): long enough to batch a realistic "handful of doc sources" corpus, short enough that stalling the daemon during an explicit, rare, user-triggered action is "annoying but not catastrophic" — and the plan explicitly directs revisiting the budget (up to ~20s) or escalating to the rejected worker-thread option if the measured cap comes back unworkably small, rather than silently shipping a useless cap. Every occurrence of the literal `2000` in the plan (Domain Glossary, Story 4.1.2, Task 4.1.2a, Task 4.1.2c) is consistently and explicitly labeled a "placeholder"/"illustrative" value pending the benchmark — verified via `grep -n MAX_CHUNKS_PER_SOURCE plan.md`, no stray "2000 as fixed fact" reference remains. Sub-batching (`SUB_BATCH_SIZE = 100`, `tokio::task::yield_now().await` between sub-batches — Task 4.1.3a) is real and correctly reasoned: it turns one worst-case 8s uninterruptible span into several ~1-2s spans the single-threaded cooperative scheduler can interleave other tool calls around. Confirmed against the actual codebase that this is a genuinely single-threaded, `!Send`, `Rc`/`RefCell`-style daemon (matches `crates/cli/src/main.rs`'s existing pattern), so `yield_now()` is a real, meaningful interleaving point, not a no-op.
- [x] Blocker 2 (concurrent interleaving) — RESOLVED — `SourceLocks` (`RefCell<HashSet<SourceId>>` + `Drop`-based RAII guard, Task 3.4.3a, `plan.md:926-957`) is acquired synchronously (no `.await` before it) at the very start of both `index_source` (Task 4.1.1a) and `remove_indexed_source` (Task 4.3.2a), *before* any HTTP fetch or file I/O — confirmed the acceptance criteria explicitly require this ("without calling `http.get` or `fs.write_file` at all"). Release is via `impl Drop for SourceLockGuard` (`self.locks.active.borrow_mut().remove(&self.id)`), so it fires automatically on every exit path — success, any `?`-propagated `Err`, early return — with no manual-cleanup-at-each-return-site risk (Rust drops local bindings unconditionally on scope exit; there is no code path that requires "remembering" to release it). `SourceId` gained `Hash` in its derive list (`plan.md:815`, confirmed the only definition site), required for `HashSet<SourceId>`. The `Rc::new(...)`-then-`.clone()`-into-closure wiring (Task 5.1.1a/b/d) is plausible and verified against the actual codebase: `crates/cli/src/main.rs:75-78` already does exactly `let http = Rc::new(NativeHttp::new()); let fs = Rc::new(NativeFs); ...`, and `NativeClock`/`ClockPort` (referenced for the new `clock` parameter) already exist in `crates/native/src/sleep.rs` and are already imported in `crates/cli/src/thin_client.rs` — the plan's claims about existing patterns are factually accurate, not invented.
- [x] Blocker 3 (disk-full/partial-write) — RESOLVED — Task 1.3.2a correctly implements temp-file-in-**same-directory** + `tokio::fs::rename` (verified against the current, actually-non-atomic `crates/native/src/fs.rs`, which does confirm the described `tokio::fs::write` truncate-then-write baseline the fix replaces). The plan explicitly states the same-directory requirement and why ("cross-filesystem renames are not [atomic]"). Concrete Given-When-Then acceptance criteria and a corresponding unit test (Task 1.3.2b) cover both the success path (no stray `.tmp-*` file left behind) and the interrupted-write path (reader never observes a torn file, old content preserved). `search_docs`'s malformed-JSONL-line handling (Task 4.2.1c) is concrete: parse `chunks.jsonl` line-by-line (not one whole-file `serde_json::from_str`), `eprintln!` a warning naming the source and 1-indexed line number, skip and continue — matching `webcrawl.rs::fetch_and_expand`'s cited skip-one-bad-item precedent, with a real Given-When-Then test (10 valid + 1 malformed line → `Ok` with 10 ranked results + one warning line).
- [x] Round-1-introduced blocker (temp-filename race) — RESOLVED — Task 1.3.2a (`plan.md:406-434`) now builds the temp filename as `format!("{path}.tmp-{}-{}", std::process::id(), TMP_FILE_COUNTER.fetch_add(1, Ordering::Relaxed))`, where `TMP_FILE_COUNTER` is a **module-level** `static AtomicU64` in `crates/native/src/fs.rs` (`plan.md:420`). This is the textbook-correct pattern: `fetch_add` is a single atomic read-modify-write whose *return value* is consumed inline (not a separate read-then-increment), it is **global**, not scoped per-path (so it doesn't degenerate back into a collision for two calls to the *same* path — the exact case that mattered for `sources.json`), and it never resets within a process's lifetime (only a process restart resets it, which also changes the PID, preserving cross-run uniqueness too). `Ordering::Relaxed` is correctly justified in the plan's own text — atomicity of `fetch_add` does not depend on the ordering parameter; only cross-variable visibility does, and no other memory needs to be ordered against this counter.
  - **Test coverage**: Story 1.3.2's third acceptance criterion (`plan.md:393-403`) explicitly requires "the two temp filenames are guaranteed distinct (not just distinct-with-high-probability)" for two overlapping `write_file` calls to the same path, and Task 1.3.2b (`plan.md:436-443`) requires that test be written (`tokio::join!` on two same-path writes, or calling the temp-path-building logic twice and asserting the two paths differ). Concrete and correctly scoped — not a vague "add a test" placeholder.
  - **Pattern Decisions explanation accuracy** (row "Chunk-file write durability", `plan.md:108-109`): accurately restates the PID-alone flaw (constant for the daemon's whole lifetime) and correctly identifies *why* `SourceLocks` doesn't cover this case — `SourceLocks` guards per-`source_id` writes to `chunks.jsonl`/`meta.json`, but `sources.json` is a single shared manifest rewritten by *every* `index_docs`/`remove_indexed_source` call regardless of which source_id it's for, so two concurrent calls for two *different* sources (each individually holding a distinct, non-conflicting `SourceLocks` guard, so neither is rejected) can still legitimately race on the one shared manifest path. This explanation is repeated consistently in three places (Pattern Decisions row 109, Task 1.3.2a's inline rationale at 412-417, and Story 1.3.2's acceptance criteria at 393-403) with no contradiction between them.

- **New — `TMP_FILE_COUNTER` missing from the Domain Glossary table** (`plan.md:64-90`). The glossary's own stated scope is "every domain term that appears as a type, method, or variable name" (`plan.md:62`), and `TMP_FILE_COUNTER` is exactly such a term (a module-level static introduced by round 2's fix), fully explained at its definition site (Task 1.3.2a) but not given a row the way `MAX_CHUNKS_PER_SOURCE`, `SUB_BATCH_SIZE`, and `SourceLocks` were. Purely cosmetic — the term is used consistently and correctly everywhere it appears; it's just not indexed in the glossary table. Trivial one-row fix if desired.
- **New — `index_docs` thin-client tool description overstates blocking** (Task 5.2.1a, `plan.md:1455`): "Indexing runs synchronously and blocks other daemon calls for its duration." This contradicts Story 4.1.3's explicit sub-batching design (Pattern Decisions "Concurrency / blocking inference" row, `plan.md:107`; Story 4.1.3 design note, `plan.md:1144-1157`), whose entire stated purpose is "the daemon can interleave other tool calls during a long indexing run instead of blocking for the entire cap's worth of chunks in one uninterruptible call." The description is accurate only in that the *calling client* blocks waiting for its own response — it overclaims that the daemon can't service other calls in the meantime, which the plan's own architecture explicitly designed against. Low-impact (errs toward an overly cautious LLM caller, not a correctness bug) — one-line text fix in Task 5.2.1a whenever it's implemented.
- Default `source_name` collision (accepted risk in the plan) is reasonably scoped given the single-user, low-source-count context. **Unchanged from round 1** — still an explicitly accepted risk in Unresolved Questions, not addressed by either fix round (reasonable to leave as-is).
